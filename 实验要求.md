# Qwen-Image StagePool / Batching 性能实验要求

## 1. 目标

在 8 卡 Ascend 910B NPU 机器上，评估 Qwen-Image PR #3208 的 encode / denoise / decode 三段拆分，以及 PR #2729 diffusion continuous batching 的性能收益。

重点回答五个问题：

1. `denoise` 使用 USP2/4/8 时，单个 DiT 实例的纵向并行收益和代价是什么。
2. 固定每个 DiT replica 使用 USP2 后，StagePool 多 replica 是否能提升吞吐。
3. `encode` / `decode` 独立或并置时，对吞吐和延迟的影响有多大。
4. `execute_model`、`step_execution no-batch`、`step_execution batching` 三条路径分别有什么性能差异。
5. 如果 USP-only 配置 OOM，TP2 + USP 叠加是否能作为兜底方案。

## 2. 固定实验条件

- 硬件：8 x Ascend 910B NPU
- 模型：`Qwen/Qwen-Image`
- 任务：text-to-image
- 主 workload：1024 x 1024，40 steps，开启 negative prompt
- 数据集：`random`
- 请求参数必须同 shape，batching 实验不要混分辨率，否则 compatible batching 不成立
- 每组先跑 smoke：`num_prompts=8, max_concurrency=2`
- 正式测试：`num_prompts=100`
- 并发 sweep：`1, 2, 4, 8, 12, 16, 24`
- 每次只改变一个维度：拓扑、`step_execution`、`max_num_seqs`
- USP sweep：`2, 4, 8`
- OOM 兜底：如果 USP-only OOM，单独测 `tensor_parallel_size=2` 叠加 USP，不和 USP-only 主结论混在一起。

## 3. 启动前置条件

Qwen-Image 模型可以通过 ModelScope 拉取和启动。启动服务前设置：

```bash
export VLLM_USE_MODELSCOPE=True
```

如果同一 shell 里连续切换多组服务配置，保持该环境变量不变即可。

## 4. 路径定义

| 路径 | denoise 配置 | 用途 |
| --- | --- | --- |
| `execute_model` | `step_execution: false`, `max_num_seqs: 1` | PR #3208 request-level denoise baseline |
| `step_no_batch` | `step_execution: true`, `max_num_seqs: 1` | PR #2729 batching baseline，量 step-wise 本身开销 |
| `step_batch2` | `step_execution: true`, `max_num_seqs: 2` | 小 batch，风险较低 |
| `step_batch4` | `step_execution: true`, `max_num_seqs: 4` | 主 batching 配置 |
| `step_batch8` | `step_execution: true`, `max_num_seqs: 8` | 可选压力测试，可能 OOM |

## 5. 并行配置规范

denoise 每个 replica 的卡数由下面的乘积决定：

```text
devices_per_denoise_replica =
  tensor_parallel_size * ulysses_degree * ring_degree * cfg_parallel_size
```

本实验默认 `ring_degree=1`、`cfg_parallel_size=1`。

### USP-only 配置

用于主实验，隔离 USP 纵向并行和 StagePool 横向并行。

```yaml
parallel_config:
  tensor_parallel_size: 1
  ulysses_degree: ${USP}
  ring_degree: 1
  sequence_parallel_size: ${USP}
  cfg_parallel_size: 1
```

### TP2 + USP 兜底配置

仅在 USP-only OOM 或显存压力导致 failed request 时使用。TP2 会改变通信和 shard 方式，结论需要单独汇报。

```yaml
parallel_config:
  tensor_parallel_size: 2
  ulysses_degree: ${USP}
  ring_degree: 1
  sequence_parallel_size: ${USP}
  cfg_parallel_size: 1
```

例如 `TP2 x USP2` 每个 denoise replica 占 4 张卡，`TP2 x USP4` 每个 denoise replica 占 8 张卡。

## 6. 实验矩阵

### A 组：单体 USP 纵向并行 baseline

目的：建立没有三段拆分、没有 StagePool 的 baseline，判断 USP2/4/8 在单体 Qwen-Image 上的单请求 latency 和吞吐变化。

| 名称 | 拓扑 | 变量 | 目的 |
| --- | --- | --- | --- |
| `A0_mono_usp2` | 2 卡单体 | USP2 | 最小单体 baseline |
| `A1_mono_usp4` | 4 卡单体 | USP4 | 看增加 USP 后单请求是否加速 |
| `A2_mono_usp8` | 8 卡单体 | USP8 | 单体最大纵向并行上限 |

启动示例：

```bash
export VLLM_USE_MODELSCOPE=True
ASCEND_RT_VISIBLE_DEVICES=0,1,2,3 \
vllm serve Qwen/Qwen-Image --omni \
  --port 8091 \
  --ulysses-degree 4 \
  --enforce-eager
```

只测 `execute_model` 路径。并发建议：`1, 2, 4, 8`。

### B 组：3-stage 单 DiT replica 的 USP 纵向并行

目的：拆成 encode / denoise / decode 后，单个 denoise replica 使用 USP2/4/8 的扩展曲线。该组用于回答“拆分后单实例 DiT 并行是否仍然有效”。

| 名称 | encode | denoise | decode | 目的 |
| --- | --- | --- | --- | --- |
| `B0_3s_usp2_r1` | NPU0 | NPU2,3, USP2 x 1 | NPU1 | 3-stage 最小 DiT baseline |
| `B1_3s_usp4_r1` | NPU0 | NPU2,3,4,5, USP4 x 1 | NPU1 | 单 replica USP4 纵向扩展 |
| `B2_3s_usp8_r1_colocate` | NPU0 | NPU0-7, USP8 x 1 | NPU0 | 单 replica USP8 上限，encode/decode 并置 |

B2 使用并置是因为 8 张卡都给 denoise。它只能说明 USP8 上限，不用于单独评估 encode/decode placement。

denoise 覆盖片段：

```yaml
runtime:
  devices: "2,3,4,5"
  num_replicas: 1
engine_args:
  step_execution: false
  max_num_seqs: 1
  enforce_eager: true
  parallel_config:
    tensor_parallel_size: 1
    ulysses_degree: 4
    ring_degree: 1
    sequence_parallel_size: 4
    cfg_parallel_size: 1
```

只测 `execute_model` 路径。并发建议：`1, 2, 4, 8, 12`。

### C 组：固定 USP2 的 StagePool 横向扩展

目的：固定每个 denoise replica 为 USP2，只改变 replica 数，评估 StagePool scale-out。该组是 PR #3208 最重要的吞吐实验。

| 名称 | encode | denoise | decode | 目的 |
| --- | --- | --- | --- | --- |
| `C0_2d_1e_1d` | NPU0 | NPU2,3, 1 x USP2 | NPU1 | 3-stage 单 replica baseline |
| `C1_4d_1ed` | NPU0 | NPU1,2,3,4, 2 x USP2 | NPU0 | 低成本 scale-out，encode/decode 并置 |
| `C2_4d_1e_1d` | NPU0 | NPU2,3,4,5, 2 x USP2 | NPU1 | 对照 C1，验证并置干扰 |
| `C3_6d_1e_1d` | NPU0 | NPU2,3,4,5,6,7, 3 x USP2 | NPU1 | 主推荐配置，拓扑最清晰 |
| `C4_8d_1ed_colocate` | NPU0 | NPU0-7, 4 x USP2 | NPU0 | 最大 replica 压力测试 |

C3 是论文主表推荐项。C4 是压力测试，失败不影响主结论，需要记录 OOM、timeout 或 p99 抖动。

denoise 覆盖片段：

```yaml
runtime:
  devices: "2,3,4,5,6,7"
  num_replicas: 3
engine_args:
  step_execution: false
  max_num_seqs: 1
  enforce_eager: true
  parallel_config:
    tensor_parallel_size: 1
    ulysses_degree: 2
    ring_degree: 1
    sequence_parallel_size: 2
    cfg_parallel_size: 1
```

只测 `execute_model` 路径。并发建议：`1, 2, 4, 8, 12, 16, 24`。

### D 组：encode / decode placement ablation

目的：解释非 DiT stage 对端到端延迟的影响，避免把 encode/decode 并置造成的抖动误判成 denoise scale-out 问题。

| 名称 | encode | denoise | decode | 对比目的 |
| --- | --- | --- | --- | --- |
| `D0_4d_1ed` | NPU0 | 2 x USP2 | NPU0 | encode/decode 并置 |
| `D1_4d_1e_1d` | NPU0 | 2 x USP2 | NPU1 | encode/decode 分离 |
| `D2_6d_1ed_1e` | NPU0,1 | 3 x USP2 | NPU0 | 增加 encode replica |
| `D3_6d_2e_2d` | NPU0,1 | 3 x USP2 | NPU0,1 | encode/decode 都 scale-out |

只在 C 组结果显示 encode/decode stage duration 明显时跑 D2/D3。否则 D0/D1 足够。

### E 组：batching 路径 ablation

目的：在同一个拓扑下比较 `execute_model`、`step_no_batch` 和 batching，区分 step-wise 开销与 continuous batching 收益。

默认选择 C3：`6D + 1E + 1D, 3 x USP2`。

| 名称 | denoise path | `max_num_seqs` | 目的 |
| --- | --- | --- | --- |
| `E0_exec` | `step_execution: false` | 1 | request-level baseline |
| `E1_step_no_batch` | `step_execution: true` | 1 | step-wise no-batch baseline |
| `E2_step_batch2` | `step_execution: true` | 2 | 小 batch 收益 |
| `E3_step_batch4` | `step_execution: true` | 4 | 主 batching 配置 |
| `E4_step_batch8` | `step_execution: true` | 8 | 压力测试，可能 OOM |

并发建议：`4, 8, 12, 16, 24`。低并发下 batching 收益可能不明显，重点看高并发吞吐和 p95/p99。

### F 组：TP2 + USP 兜底实验

目的：如果 USP-only OOM，验证 TP2 shard 后能否跑通，并记录它相对 USP-only 的性能代价。该组是稳定性兜底，不作为主加速结论。

| 名称 | encode | denoise | decode | 触发条件 | 目的 |
| --- | --- | --- | --- | --- | --- |
| `F0_tp2_usp2_r1` | NPU0 | NPU2,3,4,5, TP2 x USP2 x 1 | NPU1 | USP2 单 replica OOM 或显存过高 | 验证 TP2 是否缓解显存 |
| `F1_tp2_usp2_r2_colocate` | NPU0 | NPU0-7, TP2 x USP2 x 2 | NPU0 | 需要 2 个大 replica | 最大 2-replica 兜底 |
| `F2_tp2_usp4_r1_colocate` | NPU0 | NPU0-7, TP2 x USP4 x 1 | NPU0 | USP4/USP8 OOM | 单 replica 大并行兜底 |

denoise 覆盖片段：

```yaml
runtime:
  devices: "2,3,4,5"
  num_replicas: 1
engine_args:
  step_execution: false
  max_num_seqs: 1
  enforce_eager: true
  parallel_config:
    tensor_parallel_size: 2
    ulysses_degree: 2
    ring_degree: 1
    sequence_parallel_size: 2
    cfg_parallel_size: 1
```

记录要求：

- OOM 前的 USP-only 配置是什么。
- TP2 + USP 是否跑通。
- 显存峰值下降多少。
- QPS 和 latency 相比 USP-only 或较小 USP baseline 的变化。

## 7. Benchmark 命令模板

每个配置启动服务后，按并发 sweep 跑：

```bash
python benchmarks/diffusion/diffusion_benchmark_serving.py \
  --backend vllm-omni \
  --base-url http://localhost:8091 \
  --model Qwen/Qwen-Image \
  --task t2i \
  --dataset random \
  --num-prompts 100 \
  --max-concurrency ${C} \
  --request-rate inf \
  --warmup-requests ${C} \
  --warmup-concurrency ${C} \
  --enable-negative-prompt \
  --random-request-config '[{"width":1024,"height":1024,"num_inference_steps":40,"weight":1}]' \
  --output-file results/${EXP_NAME}_c${C}.json
```

建议并发：

```bash
for C in 1 2 4 8 12 16 24; do
  # run benchmark command
done
```

## 8. 必须记录的指标

每个 JSON 至少记录：

- `throughput_qps`
- `latency_mean`
- `latency_p50`
- `latency_p95`
- `latency_p99`
- `stage_durations_mean`
- `stage_durations_p50`
- `stage_durations_p99`
- `peak_memory_mb_max`
- `completed_requests`
- `failed_requests`

额外记录：

- 每个 stage 的 NPU 卡位
- denoise replica 数
- `tensor_parallel_size`
- `step_execution`
- `max_num_seqs`
- `ulysses_degree`
- `sequence_parallel_size`
- 每个 denoise replica 的卡数
- 是否有 OOM / timeout / failed request
- `npu-smi` 观察到的显存峰值和利用率

## 9. 结论写法要求

推荐按六层结论写：

1. **单体 baseline**：A0/A1/A2 说明 USP2/4/8 在未拆 stage 时的纵向并行趋势。
2. **拆分后 USP 纵向扩展**：B0/B1/B2 说明 3-stage 之后单个 denoise replica 的 USP2/4/8 曲线。
3. **StagePool 横向扩展**：C0/C2/C3/C4 说明固定 USP2 后增加 denoise replica 的 QPS、per-NPU QPS 和 p95/p99 变化。
4. **encode/decode placement**：C1/C2/D2/D3 说明非 DiT stage 是否影响端到端尾延迟。
5. **Batching 收益**：E0-E4 说明 `step_no_batch` 的额外开销和 `step_batch2/4/8` 的吞吐收益。
6. **OOM 兜底**：F0-F2 只在 USP-only 失败时汇报，重点说明 TP2 + USP 是否跑通、显存下降多少、性能代价多少。

PR 结论不要只写端到端 latency。必须同时给 stage duration，证明收益来自 denoise replica / batching，而不是客户端并发或 encode/decode 波动。
